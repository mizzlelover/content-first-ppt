# 03 · 确认受众（Audience）

> **必须主动向用户确认受众**，不要假设。同一份材料，受众不同，组织方式完全不同。

## 必问清单（一次性问清，给选项让用户好回答）
1. **受众是谁？**（可多选/主次）
   - 内部：分管领导 / 大老板（一把手） / 部门经理
   - 集团领导层
   - 外部第三方领导层
   - 外部客户经办（执行层）
   - 技术专家 / 评审
2. **场合与时长？**（评审/路演/汇报/谈判；几分钟）
3. **受众对该主题的专业度？**（不了解 / 大致了解 / 很专业）
4. **你想促成什么决定/动作？**（要资源？要批准？要合作？要采购？）
5. **受众最关心什么、最忌讳什么？**（如有）

## 侧重矩阵（据此设定权重，来自调研）
| 受众 | 最关心 | 弱化 | 开场落点 | 证据偏好 |
|---|---|---|---|---|
| 大老板 / 集团领导 | 战略契合、格局/高度、能带来的资源、风险与回报 | 实现细节 | 结论 + 诉求 | 关键数字、对标、一页全貌 |
| 分管领导 | 与本条线目标/KPI 的关系、可控性、协同 | 底层技术 | 与其职责挂钩的价值 | 里程碑、责任边界 |
| 部门经理 | 落地路径、排期、人力/预算、跨部门协作 | 宏大叙事 | 计划与分工 | 甘特/清单/RACI |
| 外部第三方领导 | 合作模式、互利、品牌/合规、长期关系 | 内部流程 | 合作框架 + 共赢 | 案例、信任凭证 |
| 外部客户经办 | 怎么用、解决我什么问题、上手成本、支持 | 战略愿景 | "对你有什么用" | 操作演示、流程图、FAQ |
| 技术专家 | 可行性、架构、如何实现、性能边界 | 营销话术 | 全貌框架→细节 | 架构图、指标、取舍说明 |

> 规律：**下层/执行**关注"如何实现、有何能力"；**领导层**关注"能带来什么资源、什么高度"。

## 受众视角的"问题清单"（headline 的种子）
确认受众后，立刻列出**该受众真正要判断、会质疑、要拍板的问题**（连同他的顾虑）——这是各页「问题导向标题」的种子，**不是把章节名加问号**。按受众示例：
- **监管 / 审批方（如海关）**：能否既管得住又放得开？合规吗？风险（走私 / 免税套利 / 串货 / 舆情）可控吗？品类划分依据稳吗？场地与监管仓达标吗？有先例吗？**到底要我批 / 支持什么**？
- **大老板 / 集团领导**：对战略与全局意味着什么？要我给什么资源？风险与回报如何？
- **投资人 / 外部第三方领导**：凭什么是你？护城河在哪？回报与退出？
- **技术评审**：架构成立吗？性能与边界？怎么实现？
把这些问题映射到页面，即得到高质量的「问题导向标题」（见 `05-authoring.md` 问题质量关）。

```jsonc
  "emphasize": ["按矩阵列出的重点"],
  "downplay":  ["要弱化的"],
  "opening": "开场落点（结论先行 or 先讲是什么）",

  "taboos": ["忌讳/敏感点"]
}
```

## 放行条件
- 受众、专业度、诉求**已与用户确认**；`audience-profile.json` 写好。
