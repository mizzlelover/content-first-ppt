# 04 · 问题导向编排（Narrative）

> 目标：基于 `source-model` + `audience-profile`，把内容编排成**问题导向、直奔主题**、顺序贴合受众的叙事。

## 第一原则：开场先消除信息不对称
- 第 1 张内容页（封面之后）**必须**用**最通用的语言**一句话讲清：**这是什么、要做什么**。
- 面向非专业受众时，用 `glossary` 里的平民化解释，别上来就抛术语。

## 叙事方法论
- **以受众的问题树为骨架（Minto / McKinsey）**：先画 issue tree——顶层总问题、二层 MECE 分支、三层页面问题；据此定页定序，而非套通用章节再加问号。页序遵循"父问题由子问题回答、前页是后页的必要前提"。
- **SPIN 配重（Rackham，35,000 次销售拜访实证）**：少量 Situation 快速对齐 → 立即进入 Problem → **篇幅重心放在 Implication（不解决的代价）与 Need-payoff（解决的价值）** → 最后才方案。现状页过多会稀释紧迫感。
- **宏观 ABT（Randy Olson）**：整套 deck 走 And（共识）— But（冲突/转折）— Therefore（结论/方案）。
- **Bloom 认知坡道**：理解/分析 → 评价/取舍 → 方案/创造；对董事会/投资人/监管等高成熟受众，多数页直接处于分析/评价层，别从"是什么"讲起。
- **BLUF（审批/监管受众尤其）**：早早让对方知道"**请你支持/批复什么**"，再逐条论证可行、合规、可控、有先例；不要把诉求拖到最后。

## 顺序随受众变（选模板，再微调）
**A. 不熟悉/混合/执行层（先建立认知）**
1. 是什么/做什么（消除信息差）→ 2. 为什么现在 → 3. 解决什么痛点 → 4. 怎么做 → 5. 优势 → 6. 为什么是你 → 7. 怎么合作 → 8. 商业模式
（顺序与取舍按 `audience-profile.emphasize/downplay` 调整）

**B. 专业/高层（结论先行、一步到位给全貌）**
1. 结论 + 诉求（BLUF：要什么、能带来什么）→ 2. 全貌/框架（要构建的模式、怎么合作）→ 3. 能实现的目标 → 4. 关键差异/优势 → 5.（按需）细化与证据
- 高层时间短：把"怎么实现"等细节默认放**附录**，被问到再展开。

## 把 source-model 映射到叙事节
- 每个叙事节挑选支撑它的 `facts/metrics/claims`；**每个主张都要有证据**。
- 证据偏好按 `audience-profile.evidence_pref`（领导→数字/对标/全貌；技术→架构/指标；客户→流程/案例）。
- `open_questions` 中影响主线的缺口：要么阶段③已问用户补齐，要么在叙事中**回避或诚实标注**，不要编造。

## 正文 vs 附录（这是关键差异化）
进入**附录**的内容：
- 深度细节（技术实现、详细数据、合同条款…）
- "可能被打断/被要求跳过"的部分
- 正文放不下、但被问到要能立刻调出的料

做法：
- 在正文相关页加 `links: [{label:"展开：…", target:"<附录页id>", type:"detail"}]`；
- 把附录页 `kind` 设为 `appendix` 并登记到 `deck.appendix`；
- 渲染器会自动给附录页加"返回"，讲完跳回原位。

## 面包屑/章节
- 用 `sections` 定义顶层提纲（就是面包屑的内容）；为每张正文页设 `section`。
- 章节数建议 4–7 个，名字短（如：是什么 / 为什么现在 / 怎么做 / 优势 / 合作）。

## 产物：outline（章节 + 每页骨架）
为每页先定：`section`、要点（断言句草稿）、用哪些证据、是否有"展开"链接。然后进入写页。

## 放行条件
- 顺序贴合受众；开场已消除信息差；每个主张有证据；深/可选内容已分流到附录并建好跳转。

> 完成后进入 `05-authoring.md`。
